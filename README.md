# dbhelper

A database linter, schema diff, and optimization tool. Supports Postgres and MySQL.

dbhelper reads a project configuration file that describes your database landscape — which engines, schemas, ORMs, and migration directories are in play — then parses migrations, detects conflicts, diffs against previous state, and lints for issues.

It works with migrations generated by [drizzle-kit](https://orm.drizzle.team/kit-docs/overview), [Alembic](https://alembic.sqlalchemy.org/), or raw SQL files.

## Status

Early development. The project structure and integration generators are in place; core logic is being built out. See [open issues](https://github.com/Pseudo-Development/dbhelper/issues) for the roadmap.

## Quick Start

```bash
# Build
cargo build

# Create a config file (or copy the example)
cp dbhelper.example.toml dbhelper.toml

# Validate your config
cargo run -- check dbhelper.toml

# Diff schemas (compare current migrations against last known state)
cargo run -- diff dbhelper.toml

# Lint schemas
cargo run -- lint dbhelper.toml

# Optimize schemas
cargo run -- optimize dbhelper.toml
```

## Configuration

dbhelper is driven by a TOML config file (`dbhelper.toml`) that describes your database landscape:

```toml
output_dir = ".dbhelper/state"

[[databases]]
name = "myapp"
engine = "postgres"

# Drizzle manages the public schema
[[databases.sources]]
orm = "drizzle"
migrations = "apps/web/drizzle/migrations"

# Alembic manages the analytics schema
[[databases.sources]]
orm = "alembic"
migrations = "services/analytics/alembic/versions"
schema = "analytics"
```

### Key Concepts

- **Multiple ORMs, one database**: Different teams or services can push migrations to the same database. dbhelper detects table/column name conflicts across sources.
- **Schema targeting**: Each source can target a specific schema (defaults to `public` on Postgres). Multiple sources can target the same schema — dbhelper will flag conflicts.
- **State diffing**: `dbhelper diff` parses all migration sources, builds the resulting schema, and compares it against the last known state saved in `output_dir`. This shows you what changed since the last run.
- **Schema-qualified migrations**: Migrations can be schema-qualified or not. Unqualified migrations default to the schema specified in the source config (or `public` for Postgres).

See [`dbhelper.example.toml`](dbhelper.example.toml) for the full configuration reference.

## Architecture

```
dbhelper/
├── crates/
│   ├── dbhelper-core/           # Config, schema types, diff engine, lint, optimization
│   ├── dbhelper-postgres/       # Postgres introspection via sqlx
│   ├── dbhelper-mysql/          # MySQL introspection via sqlx
│   ├── dbhelper-container/      # Ephemeral test DB containers (testcontainers)
│   └── dbhelper-cli/            # CLI binary (clap)
├── integrations/
│   ├── drizzle/                 # Drizzle-kit schema definitions → SQL fixtures
│   └── sqlalchemy/              # SQLAlchemy/Alembic models → SQL fixtures
└── tests/fixtures/              # SQL migration fixtures for testing
```

### Crates

| Crate | Purpose | Key deps |
|-------|---------|----------|
| `dbhelper-core` | Config parsing, DB-agnostic schema types, diffing, linting, optimization | serde, toml, thiserror |
| `dbhelper-postgres` | Introspect Postgres `information_schema` / `pg_catalog` | sqlx (postgres) |
| `dbhelper-mysql` | Introspect MySQL `information_schema` | sqlx (mysql) |
| `dbhelper-container` | Spin up ephemeral Postgres/MySQL containers for testing | testcontainers |
| `dbhelper-cli` | CLI interface wiring everything together | clap, tokio |

### Integration Generators

The `integrations/` directory contains non-Rust projects that define comprehensive database schemas using popular ORMs. These generate SQL migration files used as test fixtures, and they define the full feature set dbhelper must support.

**Drizzle** (TypeScript):
```bash
cd integrations/drizzle
npm install
npm run generate        # generates migrations/postgres/ and migrations/mysql/
```

**SQLAlchemy** (Python):
```bash
cd integrations/sqlalchemy
pip install -e .
alembic revision --autogenerate -m "initial"
alembic upgrade --sql head    # emit SQL without running it
```

## Supported Features

| Feature | Postgres | MySQL |
|---------|----------|-------|
| Column types (20+) | smallint through macaddr | tinyint through varbinary |
| Primary keys (single + composite) | Yes | Yes |
| Foreign keys (simple + composite) | Yes | Yes |
| Self-referencing FKs | Yes | Yes |
| FK actions (cascade, set null, restrict) | Yes | Yes |
| Unique indexes (single + composite) | Yes | Yes |
| Partial/filtered indexes | Yes | N/A |
| Check constraints | Yes | Yes (MySQL 8.0+) |
| Named enums | `CREATE TYPE` | Inline `ENUM(...)` |
| JSON/JSONB | Yes | JSON |
| Named schemas | Yes (`analytics`) | N/A |
| Unsigned integers | N/A | Yes |
| `ON UPDATE CURRENT_TIMESTAMP` | N/A | Yes |
| Binary types | bytea | `BINARY`/`VARBINARY` |
